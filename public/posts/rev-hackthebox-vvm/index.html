Rev: HTB vvm

2025-02-17yocvito
#reverse  #vm  #elf 

We have to reverse the vvm binary, an ELF 64 bits for Linux where symbols have been removed (stripped).

$ file vvm
vvm: ELF 64-bit LSB shared object, x86-64, version 1 (SYSV), dynamically linked, interpreter /lib64/ld-linux-x86-64.so.2, BuildID[sha1]=b1aeaf722aff251a97a1b0b3af16fc84039209af, for GNU/Linux 3.2.0, stripped

Searching for non encrypted/encoded strings in the binary only gives us the software version as interesting matter:

$ strings vvm
...
=======================
vvm v0.0.3
...

Additionnaly, we don’t find any embedded file in the binaryusing binwalk.

main function sequentially call 3 functions:

  • sub_2940: just prints the software banner
  • sub_1540: lot of stuff happening here, seems to set up things for later use
  • sub_28A0: this function only iterates over a global byte array and calls dynamically allocated function pointers based on current value of global array

By navigating a bit around the last 2 functions, we uncover that the binary likely obfuscate its logic with a custom VM. The sub_1540 function is responsible to setting up the vm state, instructions handlers, etc… and the sub_2810 is responsible for running the custom bytecode.

__int64 __fastcall main(__int64 a1, char **a2, char **a3)
{
  vvm_banner(); // sub_2940
  vvm_init();   // sub_1540
  vvm_run();    // sub_28A0
  return 0LL;
}

Reversing the VM setup#

The vvm_init function does all the initialization for the VM. Actually, it allocates a global heap variable that will hold all the VM instructions function handler pointers.

Some instruction handlers are part of the initial codebase and there is nothing particular vvm_init does for them retrieving the function pointer and setting it in the global handlers.

For other instruction handlers, the program decrypts the code from memory (encrypted with vigenere), creates an mmap-ed region where it stores the code, and assignates the relevant global handler to the mmaped area address.

one function is responsible for decrypting stuff and mmap-ing it

we have different types of handlers:

  • control flow related handlers (modify bc pointer)
  • memory related handlers (interact with memory without modifying bc pointer)
